# System Purpose และ Success Criteria — MaliPang

อัปเดต: 23 กรกฎาคม 2026

## จุดประสงค์สูงสุด

สร้างระบบหลังบ้านที่ช่วยให้ร้านมะลิปังรับข้อมูลจาก LINE OA แล้วเปลี่ยนเป็นข้อมูล Attendance, Payroll และ Expense ที่เชื่อถือได้ โดยเจ้าของร้านไม่ต้องคอยคัดลอกข้อมูลหรือแก้ปัญหาซ้ำทุกวัน

ระบบต้องช่วยลดงาน ไม่ใช่สร้างงานดูแลเพิ่ม

## ผู้ใช้หลัก

### พนักงาน

ต้องสามารถ:

- ส่งรูปลงเวลาง่าย
- ได้รับผลตอบกลับที่เข้าใจได้
- รู้วิธีแก้เมื่อรูปไม่ผ่าน
- ไม่ต้องเข้า Google Sheets หรือระบบเทคนิค

### เจ้าของร้าน

ต้องสามารถ:

- ตรวจความผิดปกติรายวันได้เร็ว
- เห็นยอดค่าแรงและค่าใช้จ่ายที่ตรวจย้อนหลังได้
- แก้ไขข้อมูลโดยมี audit trail
- รู้ว่ารายการใดต้องจัดการ
- เปิด/ปิดฟีเจอร์หรือ rollback ได้โดยไม่ต้องแก้ข้อมูลจำนวนมากด้วยมือ

### ผู้พัฒนา/Codex

ต้องสามารถ:

- เข้าใจ Architecture จากเอกสาร
- แก้หนึ่งงานโดยไม่ทำลายส่วนอื่น
- รัน Test และเปิด PR ได้ตามมาตรฐาน
- ไม่ต้องเดากติกาธุรกิจของร้าน

## Success Criteria ด้านหน้างาน

- พนักงานส่ง Attendance ได้โดยขั้นตอนไม่ซับซ้อน
- LINE แจ้งผลสำเร็จหรือสาเหตุที่ไม่ผ่านอย่างชัดเจน
- เจ้าของใช้เวลาตรวจระบบประจำวันไม่เกินภาระที่สมเหตุสมผล
- ปัญหาแต่ละรายการมี Trace ID/Message ID หรือหลักฐานให้ตามได้
- ไม่ต้องลงข้อมูลเดียวกันสองระบบด้วยมือ

## Success Criteria ด้านข้อมูล

- Lost event = 0
- Duplicate attendance = 0
- Duplicate payroll = 0
- Duplicate finalized expense = 0
- Attendance accuracy ≥ 99% จากชุด UAT จริง
- รายการที่ยังไม่ยืนยันไม่ถูกลงบัญชี
- D1, R2 และ Google Sheets สามารถตรวจเชื่อมโยงกันได้
- Admin correction ทุกครั้งมี audit trail

## Success Criteria ด้านธุรกิจ

- ค่าแรงไม่ผิดเพราะระบบบันทึกซ้ำหรือ IN/OUT สลับ
- ค่าใช้จ่ายไม่ถูกลงก่อนผู้ใช้ยืนยัน
- เจ้าของเห็นข้อมูลที่ใช้ตัดสินใจได้โดยไม่ต้องพึ่งนักพัฒนาทุกวัน
- ค่าใช้บริการ Cloudflare/OpenAI อยู่ในระดับที่ร้านยอมรับได้
- การเพิ่มฟีเจอร์ใหม่ไม่ทำให้ความเสี่ยงและเวลาบำรุงรักษาเพิ่มแบบควบคุมไม่ได้

## Success Criteria ด้านการดูแลระบบ

- ระบบมี health/readiness check
- Queue, DLQ และ failed jobs ตรวจได้
- มี reconcile/backfill
- มี known-good commit และ rollback plan
- Secrets ไม่อยู่ใน GitHub
- Production changes มี Approval record
- Legacy Apps Script ถูกปิดอย่างมี inventory และ backup ไม่ใช่ลบโดยเดา

## สิ่งที่ไม่ใช่เป้าหมายในระยะนี้

- ไม่สร้างระบบบัญชีเต็มรูปแบบแทนนักบัญชี
- ไม่ให้ AI ตัดสินค่าแรงหรือยอดเงินโดยไม่มี deterministic validation
- ไม่ย้ายทุกระบบของร้านเข้ามาใน Backend เดียวทันที
- ไม่เพิ่ม microservices หลายตัวโดยไม่มีประโยชน์ชัดเจน
- ไม่ปิด Apps Script ก่อน V5.2 ผ่าน Production soak
- ไม่ทำ dashboard สวยก่อน reliability และข้อมูลถูกต้อง

## ลำดับความสำคัญ

1. ความถูกต้องของข้อมูล
2. ไม่สูญหายและไม่ซ้ำ
3. ตรวจย้อนหลังและกู้คืนได้
4. ใช้งานง่ายที่หน้าร้าน
5. ความเร็ว
6. ต้นทุน
7. ฟีเจอร์เพิ่มเติม

หากมีการตัดสินใจที่ทำให้สองเป้าหมายขัดกัน ให้ใช้ลำดับนี้เป็นหลัก เว้นแต่เจ้าของร้านอนุมัติเปลี่ยนลำดับสำหรับกรณีเฉพาะ
